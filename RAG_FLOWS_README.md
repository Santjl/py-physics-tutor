# RAG Flows README

This document separates the two retrieval paths that currently exist in the project:

1. Local retrieval from the `chunks` table + Gemini/Ollama generation
2. Google retrieval (Agent Search / Discovery Engine) + Gemini generation

Both paths are connected to the main feedback endpoint through a retrieval-provider switch.

## 1. Current production flow in this repo

This is the flow currently used by `POST /attempts/{attempt_id}/feedback`.

### Summary

- PDF content is ingested into the local database
- Text is split into chunks
- Chunks are embedded and stored in the `chunks` table
- Feedback retrieval searches those local chunks
- The LLM generates the explanation using those retrieved chunks

### Main files

- `app/rag/processing.py`
- `app/rag/retrieval.py`
- `app/rag/feedback.py`
- `app/rag/feedback_builder.py`
- `app/rag/feedback_prompts.py`

### End-to-end flow

1. Admin uploads a PDF through `/documents/upload`
2. The backend extracts text and stores local `Chunk` rows in the database
3. Each chunk contains fields such as:
   - `filename`
   - `page`
   - `text`
   - `chapter_title`
   - `section_title`
   - `chunk_type`
4. When a student asks for feedback, `app/rag/feedback.py` builds a retrieval query per wrong answer
5. `app/rag/retrieval.py` retrieves relevant local chunks
6. The LLM generates feedback using those retrieved chunks as context
7. `app/rag/feedback_builder.py` converts the retrieved/cited chunks into student-facing response fields

### Where `study` comes from

`study` is built from local `models.Chunk` objects.

The conversion happens in:

- `app/rag/feedback.py`
- `app/rag/feedback_builder.py`

Specifically:

- `_retrieve_per_question()` gets chunks from local retrieval
- `_process_one_question()` decides which chunks were cited or should be used as fallback
- `_build_study_groups()` groups those chunks into:
  - `filename`
  - `pages`
  - `chapter`
  - `topic`
  - `reason`

### Where `global_references` comes from

`global_references` also comes from local retrieved chunks.

Each item currently includes:

- `filename`
- `page`
- `snippet`

### Student-visible behavior

The student now receives:

- structured `study` items showing where to study in the PDF
- structured `global_references` items showing supporting references
- explicit "Onde estudar: ..." text appended to `student_feedback` and `study_suggestion`

## 2. Google generation in the current repo

The project already supports Google for:

- chat generation
- embeddings

### Configuration

This path is controlled by:

- `LLM_PROVIDER=google`
- `EMBED_PROVIDER=google`

Main files:

- `app/rag/google_client.py`
- `app/rag/feedback.py`

### Supported auth modes

You can use either:

1. Vertex AI authentication
2. Gemini API key authentication

#### Vertex AI

Required environment:

- `GOOGLE_CLOUD_PROJECT`
- `GOOGLE_CLOUD_LOCATION` (default `us-central1`)

And local auth:

```powershell
gcloud auth application-default login
```

#### Gemini API

Required environment:

- `GOOGLE_GENAI_API_KEY`

### What this means today

If you keep the current main feedback endpoint and set the providers to `google`, then:

- embeddings are generated by Google
- LLM feedback is generated by Gemini
- retrieval still happens from the local `chunks` table

So this is not "full Google retrieval". It is local retrieval with Google models.

## 3. Google retrieval flow in the main feedback endpoint

Google retrieval is available in the main `POST /attempts/{attempt_id}/feedback` flow when `RETRIEVAL_PROVIDER=google`.

Main files:

- `app/rag/google_agent_search.py`
- `app/rag/providers.py`
- `app/rag/feedback.py`

### What Google retrieval returns

Google retrieval does not return local `models.Chunk` rows.

Instead, it returns `RetrievedContextItem`:

```python
RetrievedContextItem(
    id: str,
    title: str | None,
    source_uri: str | None,
    page_number: int | None,
    snippet: str,
    metadata: dict[str, object],
    rank: int | None,
)
```

Defined in `app/rag/providers.py`.

### Normalized fields currently extracted

`app/rag/google_agent_search.py` currently normalizes:

- `title`
- `source_uri`
- `page_number`
- `snippet`
- `metadata`

### How it is integrated

`app/rag/feedback.py` adapts `RetrievedContextItem` into temporary chunk-like objects so the existing feedback pipeline can keep using:

- prompt/source-id generation
- `study`
- `global_references`
- student-facing "Onde estudar" text

## 4. What would change if the project migrates to Google retrieval

When the main feedback flow uses Google retrieval, then:

- `study` would no longer come from local `Chunk` rows
- `study` would need to be derived from `RetrievedContextItem`
- `global_references` would also need to be derived from `RetrievedContextItem`

### Expected mapping

A practical mapping would be:

- `StudyItem.filename` <- `title`
- `StudyItem.pages` <- `page_number`
- `Citation.filename` <- `title`
- `Citation.page` <- `page_number`
- `Citation.snippet` <- `snippet`

Optional extra fields such as chapter/section can be mapped from `metadata` if those values are present in Discovery Engine results.

## 5. Can Google retrieval return chapter or section?

Possibly, but only if that metadata exists in the indexed documents and is returned by Google.

Today, the normalized object does not expose a dedicated `section` field. If section/chapter exists, it would most likely be inside:

- `metadata["struct_data"]`
- `metadata["derived_struct_data"]`

So:

- page + document title is already a realistic target
- chapter/section is possible, but depends on indexed metadata and an explicit mapping step

## 6. Setup checklist for the current working Google-backed flow

This is the path you can use right now without extra code changes.

### Environment

```env
APP_ENV=dev
DATABASE_URL=postgresql+psycopg://postgres:postgres@localhost:5432/quiz_db
LLM_PROVIDER=google
EMBED_PROVIDER=google
RETRIEVAL_PROVIDER=local
GOOGLE_CLOUD_LOCATION=us-central1
GOOGLE_CHAT_MODEL=gemini-2.5-flash
GOOGLE_EMBED_MODEL=text-embedding-005
```

Choose one auth mode:

Vertex AI:

```env
GOOGLE_CLOUD_PROJECT=your-project-id
```

Gemini API:

```env
GOOGLE_GENAI_API_KEY=your-api-key
```

### Local run

```powershell
python -m venv .venv
. .venv\Scripts\activate
pip install -r requirements.txt
alembic upgrade head
uvicorn app.main:app --reload
```

### Runtime behavior

- upload PDF
- generate local chunks
- local retrieval searches those chunks
- Gemini generates the feedback

## 7. Setup checklist for Google retrieval in the main feedback flow

This path is now available through a retrieval-provider switch.

### Environment

```env
APP_ENV=dev
DATABASE_URL=postgresql+psycopg://postgres:postgres@localhost:5432/quiz_db
LLM_PROVIDER=google
EMBED_PROVIDER=google
RETRIEVAL_PROVIDER=google
GOOGLE_CLOUD_LOCATION=us-central1
GOOGLE_CHAT_MODEL=gemini-2.5-flash
GOOGLE_EMBED_MODEL=text-embedding-005
GOOGLE_DISCOVERY_SERVING_CONFIG=default_search
GOOGLE_AGENT_SEARCH_PAGE_SIZE=4
```

Choose one Discovery Engine target:

```env
GOOGLE_DISCOVERY_ENGINE_ID=your-engine-id
```

or:

```env
GOOGLE_DISCOVERY_DATA_STORE_ID=your-data-store-id
```

And choose one auth mode:

Vertex/ADC:

```env
GOOGLE_CLOUD_PROJECT=your-project-id
GOOGLE_CLOUD_PROJECT_ID=your-project-id
GOOGLE_APPLICATION_CREDENTIALS=C:\path\to\service-account.json
```

Gemini API auth only is not enough for Discovery Engine search. Google retrieval requires Google Cloud credentials with access to Discovery Engine.

### Runtime behavior

- question context is sent to Google Agent Search
- Google returns `RetrievedContextItem`
- the backend adapts those results into chunk-like context
- the existing feedback pipeline builds `study` and `global_references` from that adapted context
- the feedback response shape remains unchanged for the frontend

## 8. Remaining limitation

Google retrieval is now integrated into the main feedback flow, but exercise retrieval is still local-only in concept. In the Google path, the backend currently returns theory context and leaves `similar_question` dependent on what the LLM infers from those theory results.
